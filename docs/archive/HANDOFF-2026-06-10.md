## What this project is

**Wallet360** is a Portuguese personal-finance web app built for personal use (and
potentially as a product with a specific wedge — see market feedback below). It
covers mortgages/credits, investments, and budget tracking. All user-facing
strings are **Portuguese (Portugal)**; code and comments are in English.

## All modules built (Phases 1–9 + recent)

### Phase 1 — Auth
Email/password signup/login/logout. Route guards. Session cookie (`connect.sid`).

### Phase 2 — Loan / Crédito (multi-credit since Jun 10)
- Full amortization engine (prazo/prestacao modes, Euribor, TAN, spread, extra amortizations).
- **Each credit is independent** — can have casa, carro, etc. each with their own schedule, KPIs, and simulator.
- Credit selector chips on the Crédito page. Overview aggregates across all credits.
- Routes: `GET /api/loan` → all credits with schedules; `PUT /api/loan/:id`; `DELETE /api/loan/:id`; payments/amortizations/euribor/simulate all take `:loanId`.
- Nav: renamed Empréstimo → Crédito.

### Phase 3 — Investments
Assets, portfolio projection (gInc/gFY/gH sliders), allocation bar, per-asset Reforçar with 3 modes (market price, manual, cash-only), asset flows history in DB (no UI yet).

### Phase 3B/3C — Watchlist + live quotes + CAGR hints
Finnhub proxy (8 Nasdaq tickers default, customizable). Yahoo Finance CAGR pills (1/3/5/10y) in AssetModal. Both wired to stock price chart.

### Phase 4 — Settings
Account (name edit, email read-only, change password). Euribor history editor. Export JSON (full backup, schema v1).

### Phase 4B — Import + Watchlist customization + Danger Zone
Destructive JSON import (v1 + legacy "Claude Design" prototype shape auto-detected). Watchlist ticker editor. Reset data / Delete account (both require current password or Google session).

### Phase 5 — Google Sign In
GIS wired in both frontend and backend. Hidden if `VITE_GOOGLE_CLIENT_ID` unset.
**Still needs:** add `https://wallet-app-updg.onrender.com` + new Vercel URL to Google Cloud Console OAuth → "Authorized JavaScript origins", then set `GOOGLE_CLIENT_ID` (backend) and `VITE_GOOGLE_CLIENT_ID` (frontend, same value) in Render + Vercel env vars.

### Phase 6 — Yahoo Finance + Frankfurter FX
Multi-period CAGR, symbol suffix probing (`.L`, `.DE`, `.MI`…), subunit normalization (GBp, ZAc). Refresh portfolio values from live market prices.

### Phase 6C — Per-stock price chart (Trading212-style)
Click any asset row or watchlist card → modal with real price progression + range selector (1M/6M/1A/5A/Máx). Chart in native currency. Cached 15min per symbol+range.

### Phase 7A/7B — Budget (Saldo)
Incomes + fixed/variable expenses. KPI strip. Category breakdown donuts + 12-month stacked bar timeline honouring startYm/endYm. Portuguese keyword auto-categorisation dictionary (~160 entries). e-Fatura-style uncategorized banner.

### Phase 7C — Bank statement import (CSV / OFX)
File picker → review table (include checkbox, editable name, kind toggle, category). Dedup by fingerprint (kind|ym|day|amount|name). Parser handles CGD/BCP/Novo Banco/Millennium/ActivoBank layouts + OFX.

### Phase 7D — Saldo: 4 buckets + "Por classificar"
Income/expense × fixed/variable = 4 tables. Imported lines land as `pending: true` in a holding box. Classify = one PATCH sets type + pending:false → row moves to the right table automatically.

### Phase 7E — PDF statement import
`pdfjs-dist` (lazy-loaded, client-side, never leaves the browser). Positional column detection (Montante / Débito/Crédito vs Saldo). Tested on CTT (88 txns) and BCP (4 txns).

### Phase 7F — Bank connect (GoCardless scaffold) + merchant-grouped Saldo
- **Full backend flow** in `backend/src/routes/bank.ts`: `/status`, `/institutions`, `/connect`, `/sync`, `DELETE /connections/:id`.
- GoCardless Bank Account Data API (free PSD2 tier, ~2500 EU banks).
- "Ligar banco" modal with security disclaimer + featured banks (CTT, Millennium BCP, Montepio).
- **Gated on `GOCARDLESS_SECRET_ID` + `GOCARDLESS_SECRET_KEY` env vars.** Until set → modal shows "brevemente". No code change needed once keys are in.
- Saldo variable list now grouped by **merchant** (pivot style): one row per comerciante, count badge, signed total, expandable transactions with date/source/value.
- New `source String?` on Income/Expense (bank name, file name, or "Manual"). In export/import backup.

### Phase 7G — Learned classification rules
The "Por classificar" box shrinks over time — when a user classifies a merchant, a rule is saved and future imports from that merchant are auto-classified. `frontend/src/lib/merchant.ts` (keep in sync with backend normalizer).

### Phase 8 — Production deploy (Render)
PG sessions, rate-limit on auth routes, static SPA serving, CORS env-driven, `render.yaml` blueprint, Neon cold-start retry in `deploy:prep`.

### Phase 9 — Wallet360 rebrand + mobile-first redesign
Bottom navigation, hero KPIs on Overview, cashflow chart (month/year toggle), module cards, mobile-first CSS throughout. Overview KPI cards clickable (drill into each module). Icon chips, color accents, denser data cards on all pages.

## Cold market feedback summary (full text in `MARKET-FEEDBACK.md`)

Researched 2026-06-10. Key points:

- **Market is brutal** — PFM apps: 3–5%/mo churn, 71% of users churn in 90 days, 90% abandoned by day 30.
- **Portugal is occupied** — Revolut (#1), MB WAY (#2), Fintonic (Iberian PFM + open banking), BudgetBakers/Spendee (SaltEdge, 5000+ EU banks).
- **Wallet360's weaknesses** — web-only (PT is mobile-first), no brand trust, sprawling feature set, no moat.
- **Success odds** — broad B2C PFM: ~1–2%. Focused PT niche: ~10–15%. Personal tool/portfolio showcase: **~95% (already a win)**.
- **The real wedge** — no competitor answers "amortizar o crédito ou investir o dinheiro?" Wallet360 has BOTH the PT mortgage simulator (Euribor/TAN/spread) AND an investment projection engine. **Fusing them is the differentiation.** Position as "gestão de crédito + finanças para portugueses."
- **Top 7 changes (harsh):**
  1. Lead with the wedge — "crédito PT + amortizar vs investir", everything else is supporting cast
  2. Make it a PWA (installable, push, offline)
  3. Time-to-value < 2 min — first run = import statement → instant dashboard
  4. Proactive insights (not just passive tables)
  5. Own privacy as the trust play ("os teus dados são teus, processados no teu browser")
  6. Don't commercialize without legal layer (GDPR data controller, AISP reliance)
  7. Kill feature creep — every new thing must serve the credit/invest wedge

## Key architectural notes

- **Prisma `db push` (not migrate)** in prod. `deploy:prep` runs `db:push:prod` (retries once for Neon cold start). Destructive if you rename a column — take a Neon snapshot first.
- **Two schema files** — `backend/prisma/schema.prisma` (SQLite dev) + `backend/prisma/schema.prod.prisma` (Postgres prod). Keep them in sync on every model change. Also update `backend/src/routes/export.ts` and `import.ts` when the data model changes.
- **CommonJS backend** (not ESM). `ts-node-dev` in dev, `node dist/index.js` in prod.
- **Vercel: no `db push`** — `vercel-build` only generates the Prisma client. Run schema changes via `npm run db:push:prod` locally or let the Render pipeline do it.
- **In-memory caches** (quotes, FX, GoCardless token) reset on Vercel cold invocations — harmless, just occasional extra upstream calls.
- **Merchant normalization** in `frontend/src/lib/merchant.ts` — must be kept in sync with the backend normalizer for learned rules to work across both sides.
- **Same-day duplicate limit** — two identical same-day transactions with same amount and description collapse to one signature (genuinely indistinguishable from re-import). By design.
