## Known traps (the ones that bite)

- **Vercel env-var changes need a REDEPLOY to take effect.** Adding/editing a var does nothing to the running deployment until you redeploy (Deployments → ⋯ → Redeploy, or the "Redeploy" prompt on the env page). This bit us with `SMTP_PASS` on 2026-07-09 — it was set but the live deploy was 7h old, so email stayed in console-fallback until the redeploy.
- **DNS for wallet360.pt is at dominios.pt (nameservers `dns1-4.host-redirect.com`), NOT Vercel.** Vercel only holds the apex A record + www CNAME; the authoritative zone is the dominios.pt "SolutEDNS" panel (`my.dominios.pt`). Any mail/verification records (DKIM/SPF/MX/DMARC) go THERE. **That panel requires TXT record CONTENT to be wrapped in double quotes** (`"v=spf1 …"`) — an unquoted TXT fails with "Content must be quoted"; it stores names fully-qualified (`resend._domainkey.wallet360.pt`).
- **The owner's account is Google-only (no password).** So `forgot-password` for `fmarques.tomas@gmail.com` correctly sends NOTHING (by design — nothing to reset). Don't use it to test SMTP; use `/api/cron/daily?force=digest` (sends the owner their own digest) instead. Resend's Emails dashboard shows SMTP sends too, so it's the delivery source of truth.
- **Two Prisma schema files** — `schema.prisma` (SQLite dev) + `schema.prod.prisma` (Postgres prod) must stay in sync on EVERY model change. Also update `backend/src/routes/export.ts` + `import.ts`. No automated guard.
- **Prisma migrations ARE tracked now** (`backend/prisma/migrations/` was previously gitignored, which hid drift and caused fresh-clone 500s). Keep committing migrations; don't re-ignore them.
- **Pushing to `main` deploys to prod** and runs `db:push:prod` — destructive on a column rename. Take a Neon snapshot before any rename/drop. All recent schema changes are **additive**: `User.pinHash` + `WebAuthnCredential` (`add_pin_and_webauthn`), `User.isDemo`, `Expense.loanId`, `Expense.matchHint`, `PortfolioSettings.language`, **`Income.matchHint`** (`add_income_match_hint`, `9dce101`), and **`PortfolioAsset.isin`** (`add_portfolio_asset_isin`, `fad58ed`), and **`BrokerConnection`** (`add_broker_connection`, `9542e29` — new table, encrypted broker creds), and **`Income.frequency`/`Expense.frequency`** (`add_budget_frequency`, `bb6058c` — default "monthly"), and **`PortfolioAsset.source`** (default `"manual"`) + new **`ImportedTxn`** table (`add_asset_source_and_imported_txn` — reconciling import dedup), and new **`DeletionLog`** table (`add_deletion_log` — pseudonymous account-deletion audit log; **not** linked to User, **not** in export/import). All additive → `db:push:prod` adds them on deploy, no snapshot needed.
- **Windows: stop the dev backend before `prisma migrate dev` (or any `npm install`).** The running `ts-node-dev` holds the Prisma query-engine DLL open, so the client-regen step fails with `EPERM … query_engine-windows.dll.node`. Kill the backend on :4000 first (`taskkill //PID <pid> //F`), or for plain dep installs use `npm install … --ignore-scripts` (skips the prisma-generate postinstall).
- **Budget `amount` is ALWAYS the monthly-equivalent — even for non-monthly items.** `Income/Expense.frequency` is **entry/display metadata only**; the modal converts period↔monthly (`lib/budgetFrequency.ts`). So `summarize`/KPIs/timeline/donuts keep using `amount` unchanged. Don't store the per-period amount in `amount` (reintroduces the ripple this design avoids). The lump-in-one-month case is the separate one-off/13th-month feature.
- **Imported assets refresh via `qty × price`, gated on `isin`** (`refreshedValue` in `portfolio.ts`). Imported holdings have a broker-accurate qty, so they show real gain/loss immediately; manual/legacy assets keep the price-delta scaling (their qty may be a placeholder). Don't drop the `isin` gate — that reintroduces the "0 €/0,0 %" bug for imports OR breaks hand-curated values.
- **Re-importing broker history DOUBLES positions if the ORIGINAL import predated per-order dedup** (bit the owner 2026-07-09). `applyPortfolioTransactions` (`routes/portfolio.ts`) is a *delta* import: it seeds qty/cost/value from the existing holding and re-applies every order whose `externalId` isn't already in `imported_txns`. The v1 CSV import recorded no per-order dedup rows, so a later "import all history" re-applied the full ledger on top of the already-correct holdings — inflating qty, `invested` AND the reforços (flows) in lockstep across every holding (fully-sold names net to 0 and vanish, which is correct). **Recovery (do NOT full-restore — that wipes the day's other edits, and the 6h Neon PITR window is tiny): rebuild each held asset by replaying its `imported_txns` from zero** (buys add, sells reduce at running average — the same avg-cost logic the importer uses), set `qty`/`invested`/`value`(=cost basis, then "Atualizar valores") and rebuild that asset's `portfolio_flows` grouped by `ym` from the buy orders. `imported_txns` is UNIQUE on `(userId, source, externalId)`, so it holds exactly one clean copy of each order — it's the source of truth (and is what IRS mais-valias reads, so leave it intact). Verify after: per-asset `flow_sum == invested`, `side IS NULL` count = 0, `orphan_flows` = 0.
- **Optional integrations are env-gated (off until set):** `SENTRY_DSN`/`VITE_SENTRY_DSN` (monitoring), `UPSTASH_REDIS_REST_URL`+`_TOKEN` (shared rate-limit/lockout → in-memory fallback), `BROKER_ENC_KEY` (Trading 212 live-sync). **`VITE_SENTRY_DSN` is build-time** — it must be in the Vercel *build* env (Vite inlines it; with no DSN the SDK is tree-shaken out entirely). The Trading 212 **auth header is unconfirmed** — `lib/trading212.ts` `resolveAuth` tries raw/Bearer key then secret; pin it once a real key connects.
- **Statement imports are NOT UTF-8** — most PT bank CSV/OFX exports are Windows-1252/Latin-1. `ImportStatementModal` reads as ArrayBuffer, tries UTF-8, falls back to windows-1252 when it sees `�`. Don't revert to `readAsText(file, 'utf-8')` (that's what produced "SOLU��O").
- **`gFY` is "anos sem aumento"** (an int, contribution-growth delay), NOT a return %. Don't use it as the investment return — the wedge default uses the avg per-asset `expectedReturn` (`frontend/src/lib/compareDefaults.ts`).
- **Plan vs actual is derived from `source`** (`!!source` ⇒ imported actual; `null` ⇒ recurring plan). Changing where `source` is set moves rows between lanes. **But note the deliberate split across views:** the headline KPIs (Início + top of Saldo) and Análise are plan-based / planeado-vs-real, while **"Movimentos do mês" (`VariableMonths`) intentionally shows the month's *real* movements** (actuals + manual) with a real summary. Don't "unify" them by making Movimentos plan-only — that's the regression we fixed earlier.
- **`flatGross(annual)` in `simulate.ts` takes a FRACTION, not a percent** — break-even search feeds it `0..2`, so the ±1σ band must pass `… / 100` (forgetting it made `rr≈2.07` compound to ~1e119, fixed in `dde91a9`). **AND the band must scale σ by √T:** σ is a *single-year* std-dev, but the **terminal** gain's ±1σ over T years is σ·√T in log-space → shift the annualized rate by **σ/√T** (`riskVolN / Math.sqrt(horizonYears)`) then compound. Applying σ to *every* year (the original) compounds to absurd extremes — the user's real card showed a **€8.7M "Ano bom"**. Fixed this session (σ/√T, deployed).
- **PWA / installed app-icon is OS-cached — it won't refresh on its own.** Updating `public/favicon.svg` + `npm run generate-pwa-assets -w frontend` ships new icons, but an already-installed home-screen PWA keeps the old icon until it's **removed and re-added (reinstalled)** (a browser hard-refresh only updates the favicon/in-app UI). The whole icon set is generated from the **single** source `public/favicon.svg`; `pwa-assets.config.ts` overrides the minimal-2023 preset's **0.3 white padding** → **0 padding + navy background** so the tile fills edge-to-edge (else the installed icon has a white border). Mark size = the `scale(…)` in favicon.svg (≈1.9 ≈ 60% fill, kept inside the maskable safe zone).
- **No emoji as UI icons — use `Icon`/`BrandMark`.** The rebrand replaced every emoji chip with the line-icon set (`components/ui/Icon.tsx`, `StateBlock.icon` is now an `IconName`); don't reintroduce emoji as icons. `.btn` is inline-flex so `<Icon> + text` aligns. (Functional glyphs `✓ ✕ → ✎` are fine.)
- **Theme = CSS tokens + a dark override block; never hardcode a colour.** Every colour is `var(--token)`; light values live in `index.css` `:root`, dark values in `:root[data-theme="dark"]`. A new UI colour MUST be added as a token in **both** (or it won't adapt in dark mode). The nav is intentionally navy (`--ink`) in both themes. The one-time literal→token conversion used a lowercase-`:root`-defs / uppercase-rule-literals `sed` trick so a blanket replace wouldn't clobber the definitions.
- **Chart.js colours are theme-aware via `lib/chartTheme.ts`.** Canvas can't read CSS vars, so charts pull grid/axis-text/segment colours from `useChartColors()` and **must include `cc.resolved` (or the derived colours) in their options `useMemo` deps** so they redraw on theme toggle. Donut legend text needs `fontColor` **per generated item** (a custom `generateLabels` bypasses `labels.color` → falls back to Chart.js's dark default).
- **Google (GSI) button in dark mode: the white frame is cropped, not styled.** On the real domain GSI renders the button in a **cross-origin FedCM iframe** and paints a light surface in the ~10px/2px padding around the 320×40 dark button — unreachable by CSS. `index.css` clips `.gsi-button` to the content box in dark mode (`overflow:hidden` + a negative iframe margin). **The offsets are geometry-dependent** on GSI's current iframe size (`width:320, size:large`); if Google changes the button padding, re-measure. `color-scheme: dark` on the iframe does **not** help (it's already inherited and GSI ignores it).
- **The service worker caches the JS/CSS bundle** (Workbox). After a deploy, **visual/CSS changes don't appear until the SW updates** — hard-refresh (Ctrl+Shift+R), or reopen/reinstall the installed PWA. This repeatedly looked like "the fix didn't work" this session when it was just a stale cached bundle. To force it in a debugging browser: unregister the SW + `caches.delete(...)` then reload.
- **Settings is 4 grouped tabs now** (`Conta · Segurança · Preferências · Dados`), was 7 flat. The section components are unchanged — the grouping + section-labels live in `pages/Settings.tsx`. Euribor/Watchlist/Idioma sit under Preferências.
- **Classifying as "Fixed" deliberately clears `source` + `startYm`/`endYm`** (`/classify` + `/bulk-update` in `backend/src/routes/budget.ts`) — that promotion is what moves a movement into the recurring Fixas plan (`c8d6ca6`). Don't "fix" the source-clearing thinking it's a bug. `/classify` promotes only the clicked line; siblings stay actuals on purpose (avoids duplicate recurring rows).
- **Dev uses in-memory sessions** — restarting the local backend logs everyone out (re-login in the browser). Prod uses the Postgres session store, so it persists across deploys.
- **Yahoo ticker search is an unofficial endpoint** (`query1.finance.yahoo.com/v1/finance/search`). Finnhub search is the backup. It returns **no currency** — currency is derived backend-side from the symbol suffix/exchange (`currencyForResult`); it's a hint, not authoritative.
- **EUR auto-fill is EUR-only.** AssetModal's `fillPrice` uses `priceEur` (or the native price *only* when it's already EUR). If the FX lookup fails for a non-EUR listing, the amount fields stay empty on purpose — **don't reintroduce a `?? nativePrice` fallback** into the "(€)" fields, or you'll fill a USD/KRW number labelled "auto-preenchido".
- **GBp (pence) vs GBP.** LSE listings show a "GBP" badge but Yahoo reports prices in `GBp` (pence). `normalizeSubunit` in `fx.ts` divides by 100 before converting, so `priceEur` is correct — don't "fix" this by treating `GBp` as pounds in the conversion.
- **Merchant normalization** in `frontend/src/lib/merchant.ts` must match the backend normalizer or learned classification rules break.
- **Backend is CommonJS**, not ESM.
- **i18n: never hardcode UI strings** — all user-facing text goes through `react-i18next`. Add keys to BOTH `frontend/src/i18n/locales/pt/<ns>.json` AND `…/en/<ns>.json` (keep them in **parity** — there's a key-diff check + dev `missingKeyHandler`). Type-safe keys mean dynamic `t(variable)` needs the variable typed as a literal union (`as const` / explicit union), not `string`.
- **i18n categories: translate the LABEL, not the stored value.** `Income.category`/`Expense.category` persist the **canonical pt** string (`Habitação`); `categoryLabel()` maps it to the display label. An expense saved in English mode must still store `Habitação`, not `Housing`. The parsing `DICTIONARY` stays pt.
